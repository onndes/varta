# План рефакторинга архитектуры VARTA-2026

## Выявленные проблемы

### 1. **Большие компоненты**
- `ScheduleView.tsx` (1000 строк) - слишком много логики в одном файле
- `UsersView.tsx` (413 строк) - много ответственности
- `App.tsx` (346 строк) - перегружен логикой

### 2. **Отсутствие разделения слоёв**
- Бизнес-логика смешана с UI-компонентами
- Прямые обращения к БД из компонентов
- Нет кастомных хуков для переиспользования логики

### 3. **Дублирование кода**
- Функции расчета нагрузки дублируются
- Логика работы с датами разбросана
- Повторяющиеся паттерны работы с БД

### 4. **Отсутствие модульности**
- Нет сервисного слоя
- Нет хуков для работы с БД
- Компоненты тесно связаны

### 5. **Типизация**
- `any` в некоторых местах (исправлено частично)
- Нет валидации данных

## Преимущества после рефакторинга

1. **Читаемость** - меньшие файлы, понятная структура
2. **Тестируемость** - изолированная логика легко тестируется
3. **Переиспользование** - хуки и сервисы можно использовать везде
4. **Масштабируемость** - легко добавлять новые функции
5. **Поддерживаемость** - легче находить и исправлять баги

## Риски

- Большой объём изменений
- Возможность внесения багов
- Необходимость тестирования всех функций

## Рекомендации по выполнению

1. Делать рефакторинг поэтапно
2. Тестировать после каждого этапа
3. Создавать бэкапы перед крупными изменениями
4. Не менять логику, только структуру
5. Использовать Git для отслеживания изменений
