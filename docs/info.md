## 🎯 Цель приложения

VARTA — это PWA/Tauri-приложение для автоматического и полуавтоматического построения графиков дежурств в воинском подразделении. Основная задача — **справедливое распределение нагрузки** с учётом: статусов бойцов (отпуск, командировка, больничный), весовых коэффициентов дней (выходные тяжелее), кармы (долгов), несовместимости пар, несколько баз данных (подразделений).

---

## 🏗️ Архитектура

```
React + Vite (PWA / Tauri)
├── IndexedDB (Dexie.js) — хранилище данных
├── src/components/ — UI-компоненты
├── src/hooks/ — бизнес-логика (React hooks)
├── src/services/ — работа с БД и алгоритмы
└── src/utils/ — утилиты, константы, типы
```

**Хранилище (Dexie v9, 4 таблицы):**

- `users` — личный состав
- `schedule` — записи дежурств
- `auditLog` — лог действий
- `appState` — настройки (ключ-значение)

**Мультибаза:** каждое подразделение — отдельная IndexedDB. Метаданные в `localStorage`.

## 👤 Структура бойца (User)

```tsx
{
  name, rank,
  status: 'ACTIVE' | 'VACATION' | 'TRIP' | 'SICK' | 'ABSENT' | 'OTHER',
  statusFrom, statusTo,          // диапазон статуса
  isActive,                      // уволен/нет (серый в отдельной вкладке)
  excludeFromAuto,               // только ручное назначение
  isExtra,                       // стажёр/водитель — только ручное
  debt,                          // карма (число): < 0 = боец "должен" системе
  owedDays: { dayIdx: count },   // сколько раз должен именно этот день недели
  blockedDays: [1..7],           // заблокированные дни недели (ISO: 1=Пн, 7=Вс)
  blockedDaysFrom, blockedDaysTo,// период действия блокировки
  restBeforeStatus,              // день перед уходом в статус — недоступен
  restAfterStatus,               // день после возврата — недоступен
  incompatibleWith: [userId],    // нельзя на соседних днях с этими бойцами
  dateAddedToAuto,               // дата начала учёта в авто-планировщике
}
```

## 📅 Запись расписания (ScheduleEntry)

```tsx
{
  date: 'YYYY-MM-DD',
  userId: number | number[],     // один или несколько (dutiesPerDay > 1)
  type: 'auto' | 'manual' | 'replace' | 'swap' | 'history' | 'import',
  isLocked: boolean
}
```

**Типы записей важны для логики:**

- `history` / `import` — исторические, могут исключаться из расчёта нагрузки (`ignoreHistoryInLogic`)
- `manual/replace/swap` — ручные, авто-перезапись запрещена
- `auto` — могут быть перезаписаны при перерасчёте

## ⚙️ Настройки логики — AutoScheduleOptions

Это **центральный конфиг** алгоритма. Все параметры используются в `autoFillSchedule` и `buildUserComparator`:

| Параметр | Что делает | Работает? |
| --- | --- | --- |
| `avoidConsecutiveDays` | Не ставить подряд (filterByRestDays) | ✅ Да |
| `minRestDays` | Минимум дней отдыха (1 = не подряд, 2 = через день) | ✅ Да |
| `respectOwedDays` | Приоритет бойцам с owedDays для этого дня | ✅ Да |
| `considerLoad` | Учитывать нагрузку при сортировке + пост-балансировка | ✅ Да |
| `aggressiveLoadBalancing` | Принудительно переставлять при большом разрыве нагрузки | ✅ Да |
| `aggressiveLoadBalancingThreshold` | Порог разрыва для агрессивного балансирования (0.2 = 20%) | ✅ Да |
| `limitOneDutyPerWeekWhenSevenPlus` | 1 наряд в неделю при 7+ доступных | ✅ Да |
| `allowDebtUsersExtraWeeklyAssignments` | Должники могут дежурить чаще | ✅ Да |
| `debtUsersWeeklyLimit` | Максимум нарядов в неделю для должников (1–4) | ✅ Да |
| `prioritizeFasterDebtRepayment` | Ускоренное погашение долга | ✅ Да |
| `forceUseAllWhenFew` | При ≤7 бойцов — использовать всех циклически | ✅ Да |

## 🧠 Алгоритм автопланирования

### Входные данные

```
targetDates[], users[], schedule{}, dayWeights{}, dutiesPerDay, options
```

### Шаг 1: Жадный проход (autoFillSchedule → scheduler.ts)

Для каждой даты:

1. **Пропустить** прошедшие даты и locked-записи
2. **Пропустить** ручные назначения (isManualType)
3. **Сформировать пул кандидатов** — активные, не isExtra, не excludeFromAuto, доступные (`isUserAvailable`)
4. **Применить фильтры:**
    - `filterByRestDays` — убрать тех, кто дежурил в пределах minRestDays
    - `filterByWeeklyCap` — убрать превысивших лимит недели (если 7+ бойцов)
    - `filterByIncompatiblePairs` — убрать несовместимых с соседними днями
5. **Отсортировать** по `buildUserComparator`
6. **Взять первых N** (где N = dutiesPerDay - уже назначены)
7. **Записать** в tempSchedule и updates

### Шаг 2: Пост-балансировка (postBalancePass)

Итеративно (до 100 итераций):

- Считать нормализованную нагрузку каждого бойца пула
- Найти самого перегруженного и самого недогруженного
- Если разница > `REBALANCE_THRESHOLD` (3%) — попробовать переставить запись
- Проверить все ограничения для новой даты
- Если OK → переставить, иначе пропустить
- Повторять пока идут изменения

## 🏆 Компаратор приоритетов (buildUserComparator)

Порядок сравнения двух кандидатов (от важнейшего к менее важному):

```
0. Агрессивное балансирование (если включено и разрыв > threshold)
1. owedDays — должен именно этот день недели
2. Скорость погашения долга (getDebtRepaymentScore)
3. Нормализованная нагрузка по дням недели (countUserDaysOfWeek / доступных дней)
4. Количество нарядов в текущей неделе
5. Дней с последнего наряда (daysSinceLastAssignment) — кто дольше ждёт
6. Общее количество нарядов (нормализовано к доступности)
7. Взвешенная нагрузка + карма (нормализовано)
8. Math.random() — для устранения систематического смещения
```

---

## ⚖️ Система кармы (debt / owedDays)

**debt** — числовой показатель: отрицательный = боец "должен", положительный = система "должна" бойцу.

**Как изменяется:**

- Снят с наряда `reason='request'` → `debt -= dayWeight`, `owedDays[dayIdx]++`
- Снят `reason='work'` → карма не меняется
- Назначен авто на его "должный" день → `owedDays[dayIdx]--`, `debt += weight`
- Перенос на более сложный день → `debt += (toWeight - fromWeight)`
- Ручной сброс долга → `debt = 0`

**Ограничение:** `debt` ограничен диапазоном `[-maxDebt .. +∞]`

## 🔍 Проверка доступности (isUserAvailable / isHardUnavailable)

Многоуровневая проверка:

```
1. isActive=false → UNAVAILABLE
2. blockedDays содержит этот день недели → DAY_BLOCKED
   (с проверкой blockedDaysFrom/To)
3. status=ACTIVE → AVAILABLE
4. status=VACATION/TRIP/SICK/ABSENT/OTHER:
   - Без дат → всегда недоступен
   - С датами: dateStr в диапазоне [statusFrom..statusTo] → STATUS_BUSY
   - restBeforeStatus: день перед statusFrom → PRE_STATUS_DAY
   - restAfterStatus: день после statusTo → REST_DAY
   - Вне диапазона → AVAILABLE
5. Дежурил вчора → REST_DAY (проверка wasPrevDayAssigned)
```

## 🔄 Каскадный перерасчёт

Когда меняется боец или расписание → записывается `cascadeStartDate`. На панели управления появляется кнопка **Оптимізація**.

При нажатии:

1. Берёт все AUTO-записи начиная с `cascadeStartDate`
2. Удаляет их из БД
3. Запускает `autoFillSchedule` заново для этих дат
4. Сохраняет результат

Ручные записи (manual/replace/swap) **не трогаются**.

## ⚠️ Потенциальные проблемы и неточности в коде

### 1. `useAssignmentModal.ts` — незавершённые функции

`isOnRestDay`, `getTransferSourceDate`, `executeAssign` — в файле видны заглушки с неполной реализацией (видимо, в [project-all.md](http://project-all.md/) попал обрезанный вариант).

### 2. `useWeekNavigation.ts` — `shiftWeek` не добавляет дни

```tsx
const newDate = new Date(prev);
return newDate; // ← не добавляет offset!
```

Реальный код, скорее всего, имеет `newDate.setDate(newDate.getDate() + offset * 7)`.

### 3. `getWeekDates` — не добавляет `i` дней

```tsx
const d = new Date(monday); // ← i никуда не применяется
```

Аналогично — в [project-all.md](http://project-all.md/) обрезан код.

### 4. `filterByRestDays` — проверка вперёд (forward check)

Проверяет `i = 1..minRest` назад, но в тестах есть кейс "боец чергував завтра → відфільтрований" — это значит фильтр также смотрит вперёд. В коде это реализовано, но logика в helpers.ts показана частично.

### 5. `shouldEnforceOneDutyPerWeek` — переменная `schedule` не передаётся в тесте

В тестах используется `schedule` как глобальная переменная без объявления — скорее всего `{}` по умолчанию.

## 📊 Итоговая схема работы

```
Пользователь нажимает "Заповнити прогалини"
    ↓
useScheduleActions.runFillGaps()
    ↓
pushHistory() → сохранить снимок для undo
    ↓
useAutoScheduler.fillGaps(dates)
    ↓
autoFillSchedule(dates, users, schedule, ...)
    ├── для каждой даты:
    │   ├── построить пул (isUserAvailable)
    │   ├── filterByRestDays
    │   ├── filterByWeeklyCap
    │   ├── filterByIncompatiblePairs
    │   └── sortBy(buildUserComparator) → взять первого
    │       приоритеты: owedDays → debt → weekLoad → wait → totalLoad
    └── postBalancePass() → выровнять нагрузку итеративно
    ↓
saveAutoSchedule(entries, dayWeights)
    ├── db.schedule.put(entry)
    └── repayOwedDay() для каждого назначенного
    ↓
logAction('AUTO_FILL', ...)
    ↓
refreshData() → перерисовка
```
